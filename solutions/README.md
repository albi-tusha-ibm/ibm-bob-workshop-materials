# Workshop Solutions

This directory contains reference solutions for all workshop exercises. These solutions are meant to **inspire and guide**, not to be prescriptive. There are often multiple valid approaches to each problem.

## 📁 Available Solutions

### [Module 1: Application Engineering Solutions](module-1-solutions.md)
Reference solutions for the e-commerce microservices scenario, including:
- **Exercise 1:** Search & Discovery - Finding REST endpoints, communication patterns, Kafka configs
- **Exercise 2:** Log Analysis & Troubleshooting - Payment timeouts, Kafka lag, memory leaks
- **Exercise 3:** Documentation Synthesis - Data flow diagrams, documentation gaps
- **Exercise 4:** Incident Response - Postmortems, incident reports, action items
- **Exercise 5:** Code Review & Reasoning - Error handling, fraud detection, race conditions

### [Module 2: IT Infrastructure Solutions](module-2-solutions.md)
Reference solutions for the hybrid cloud infrastructure scenario, including:
- **Exercise 1:** Search and Discovery - High CPU servers, security vulnerabilities, firewall rules
- **Exercise 2:** Summarization and Analysis - Capacity trends, alert patterns, vulnerability prioritization
- **Exercise 3:** Troubleshooting - Network outages, storage capacity issues, VPN connectivity
- **Exercise 4:** Documentation and Drafting - Runbooks, incident reports
- **Exercise 5:** Reasoning and Planning - Capacity planning, patching strategy, architecture improvements

## 💡 How to Use These Solutions

### During the Workshop

1. **Try First:** Attempt each exercise on your own before looking at solutions
2. **Compare Approaches:** After completing an exercise, compare your approach with the reference solution
3. **Learn Patterns:** Identify patterns and techniques you can apply to other problems
4. **Ask Questions:** If a solution approach is unclear, ask your facilitator or Bob for clarification

### After the Workshop

1. **Review:** Go through solutions for exercises you found challenging
2. **Practice:** Try alternative approaches suggested in the solutions
3. **Adapt:** Apply these patterns to your own work scenarios
4. **Share:** Discuss interesting solutions with your team

## 🎯 What Makes a Good Solution?

Reference solutions demonstrate:

- **Multiple Approaches:** Often there's more than one way to solve a problem
- **Bob Interaction:** Example prompts that work well with Bob
- **Best Practices:** Industry-standard approaches and patterns
- **Real-World Context:** Considerations for production environments
- **Trade-offs:** Discussion of pros and cons for different approaches

## ⚠️ Important Notes

### Solutions Are Not Prescriptive

- These are **example approaches**, not the only correct answers
- Your solution may be equally valid or even better
- Real-world scenarios often require adaptation based on specific constraints
- Use these as learning tools, not as templates to copy

### Bob Can Help You Discover Solutions

Instead of just reading solutions:
- Ask Bob to help you work through the problem
- Request Bob to explain different approaches
- Have Bob compare your solution with alternatives
- Use Bob to understand why certain approaches work better

### Solutions Evolve

- These solutions reflect current best practices
- Technologies and patterns change over time
- Always consider your specific context and requirements
- Contribute improvements via pull requests (see [CONTRIBUTING.md](../CONTRIBUTING.md))

## 🤔 Common Questions

**Q: Should I look at solutions if I'm stuck?**  
A: Try asking Bob for hints first. If still stuck after 5-10 minutes, reviewing the solution approach (not the full answer) can help you get unstuck.

**Q: My solution is different from the reference. Is it wrong?**  
A: Not necessarily! There are often multiple valid approaches. Focus on whether your solution meets the success criteria and follows best practices.

**Q: Can I use these solutions in my real work?**  
A: Yes, but adapt them to your specific context. These are educational examples that may need modification for production use.

**Q: How do I know if my solution is good?**  
A: Check against the success criteria in each module's README. Also consider: Does it solve the problem? Is it maintainable? Does it follow best practices?
